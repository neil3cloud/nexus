## `NEXUS-REV-2026-07-17-006` Observation O-001 — Sprint 62 integration harness exercises the legacy caller-supplied path, not the new production auto-subscription path

It continues to validate that the legacy caller-supplied-attribution path still functions end-to-end, but does not exercise the new `initialize()`-registered production subscription or the `handleEvent`-driven `EngineeringDecisionCorrelationService`-based attribution resolution. That behavior is fully covered at the unit level instead, by `recovery-requirement.test.ts`'s `S69-1` through `S69-9` — an established pattern already certified for Sprint 68's equivalent consumer. Not a defect. No Builder Task. A future Sprint touching this integration harness MAY consider updating it to exercise the production auto-subscription path directly.

## `NEXUS-REV-2026-07-17-005` Observation O-001 — Redundant, unreachable `processedEventIds` duplicate-check

Category 6, Observation, Informational (Sprint 68; carried forward). `GovernanceGatedWorkflowAdvancementConsumer.handleEvent` maintains both a `resultsByEventId` cache (checked first, on every call, for any previously-seen `eventId`) and a separate `processedEventIds` Set (checked later, only reached after the `resultsByEventId` check has already passed). Every code path that adds to `processedEventIds` also calls `recordResult`, which populates `resultsByEventId` for the same `eventId` in the same call, so on any actual re-delivery of a previously-handled event the earlier `resultsByEventId` check always returns first — the `processedEventIds.has(...)` branch is provably unreachable. Not a correctness defect: the idempotency guarantee it exists to provide is already fully satisfied by the `resultsByEventId` cache, confirmed by test `S68-6`. No Builder Task. Not touched by Sprint 69. A future Sprint touching this consumer MAY simplify by removing the redundant `processedEventIds` Set.

## `NEXUS-REV-2026-07-17-003` Observation O-001 — Test name overstates Workflow-position lookup coverage

Category 6, Observation, Informational (Sprint 67; carried forward). Test `S67-7` ("resolves by Workflow position and fails closed on ambiguous repository state") is named as if it exercises Workflow-position lookup ambiguity, but its two constructed correlations use different `engineeringSessionId` values, so their position keys never collide; the ambiguity it actually exercises and asserts is `findByReviewId` ambiguity. Defensible, RFC-0004-consistent design choice, not a defect. No Builder Task. Not touched by Sprint 68 or 69.
